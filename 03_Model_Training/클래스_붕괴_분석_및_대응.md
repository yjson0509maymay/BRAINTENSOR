## 1. 무슨 문제가 있었나 (증상)

Base 모델(`CNN3D_Base`)을 논문 Study2 하이퍼파라미터(batch=32, lr=0.01, epoch=30)로
학습시키면, 실행할 때마다(정확히는 시드를 바꿀 때마다) **Control/Prodromal/PD 세 클래스
중 하나가 recall 0.00으로 완전히 무시되는 현상**이 반복 관찰됨.

| 실행 | Test Accuracy | 붕괴된 클래스 |
|---|---|---|
| 새 전처리(ref21order), seed=42 | 56.52% | **Control** (precision/recall/f1 전부 0.00) |
| 새 전처리(ref21order), seed=43 | 50.00% | **Prodromal** (precision/recall/f1 전부 0.00) |

두 실행 다 데이터·코드는 동일하고 `--seed`만 다름. 즉 어떤 클래스가 죽을지는
"버그"가 아니라 **시드에 따라 방향이 바뀌는 학습 불안정성**으로 판단됨.

## 2. 왜 이런 일이 생기는가 (원인 분석)

### 2-1. 구조적 배경: classifier 레이어 하나가 파라미터의 98.7%

`CNN3D_Base`는 논문이 명시한 그대로 "conv → conv → ReLU → pool → BatchNorm →
**flatten → 곧바로 Linear(SoftMax)**" 구조라, 중간에 hidden FC 레이어가 없음.

- Flatten 차원: 64채널 × 28×28×28 = **1,404,928**
- Classifier: `nn.Linear(1,404,928 → 3)` → 파라미터 **4,214,787개**
- 모델 전체 파라미터: 4,271,171개 → **classifier 하나가 98.7%**를 차지

### 2-2. 초기화 공식의 숨은 가정이 깨짐

- He(Kaiming) 초기화 공식 `std = sqrt(2/fan_in)`은 입력 1,404,928개가 서로
  **통계적으로 독립(i.i.d.)**이라는 가정 하에 "출력 분산이 적당한 크기가 되도록"
  설계된 것.
- 그런데 실제 입력은 conv+pool을 거친 3D 특징맵이라 **이웃 복셀끼리 값이 비슷함**
  (공간적 상관관계) — 이 독립성 가정이 깨짐.
- 상관된 입력의 가중합은 독립 가정보다 실제 분산이 훨씬 커짐 → **초기화 직후부터
  로짓이 이론치보다 훨씬 큰 값으로 튐**.

### 2-3. 로짓 폭주 → 손실 폭주 → 학습 궤적 불안정

- 로짓이 크면 "자신 있게 틀린" 예측에 대한 Cross-Entropy 손실이 기하급수적으로
  커짐 — 실제로 epoch 1 손실이 수천 단위로 관측됨(3클래스 정상 기준은 ln(3)≈1.1).
- 여기에 논문이 지정한 **Adam, lr=0.01**이 더해지면, Adam은 420만 개 파라미터
  각각에 거의 균일한 크기(≈lr)로 업데이트를 주기 때문에, 이 거대한 레이어가
  단 몇 스텝 만에 크게 요동침.
- 학습 샘플은 212개뿐인데 파라미터는 420만 개(**샘플당 파라미터 약 2만 개**,
  극심한 과대적합 조건) — 모델이 진짜 판별 특징을 배우기보다 "애매하면 한
  클래스로 몰아 찍는" 지름길이 평균 손실을 더 쉽게 낮추는 상황이 생김.
- 이 지름길이 바로 특정 클래스 recall 0(붕괴)으로 드러남. 어느 클래스가
  희생될지는 초기화·데이터 순서의 무작위성이 결정하므로 시드마다 달라짐.

## 3. 적용한 대응: classifier 출력 스케일 보정

### 3-1. 무엇을 바꿨나

`train_ablation.py`에 `calibrate_classifier_scale()` 함수를 추가.

1. 모델을 생성한 직후(학습 시작 전), 실제 train 배치 하나로 forward pass를
   한 번 수행해 classifier 출력(로짓)의 **실제** 표준편차를 측정.
2. 이 값이 목표치(1.0)가 되도록 `classifier.weight`/`bias`를 사후 스케일링.

```python
@torch.no_grad()
def calibrate_classifier_scale(model, sample_x, device, target_std=1.0):
    ...
    logits = model(sample_x.to(device))
    current_std = logits.std().item()
    scale = target_std / current_std
    model.classifier.weight.mul_(scale)
    model.classifier.bias.mul_(scale)
```

기본 활성화, `--no_calibrate_init` 옵션으로 끌 수 있음.

### 3-2. 왜 이게 "논문 재현 원칙 위반"이 아닌가

- 모델 구조(Flatten → Linear → SoftMax, 논문 명시)는 **전혀 바꾸지 않음** —
  가중치 크기만 사후 조정.
- lr / batch size / epoch 등 논문이 명시한 하이퍼파라미터도 **전혀 바꾸지 않음**.
- He(Kaiming) 초기화 자체가 애초에 논문에 없는, 우리가 이미 자체 결정했던
  구현 디테일(`_init_weights` 주석 참조) — 이번 조치는 그 자체 결정 사항을
  "상관된 입력이라는 실제 상황에 맞게 보정"하는 것뿐, 새로운 종류의 임의
  변경이 아님.

## 4. 효과 검증: 보정 전 vs 후 (같은 시드로 비교)

| 전처리 | 시드 | 보정 전 | 보정 후 |
|---|---|---|---|
| 새 전처리(ref21order) | 42 | Acc 56.52%, F1 46.76%, **Control 붕괴** | Acc 60.87%, F1 **61.15%**, 붕괴 없음 |
| 새 전처리(ref21order) | 43 | Acc 50.00%, F1 34.52%, **Prodromal 붕괴** | Acc 52.17%, F1 44.03%, 붕괴 없음(Prodromal 약함: 0.33/0.12/0.18) |
| 원본 전처리(0713_v2) | 42 | Acc 47.83%, F1 43.68% | Acc 52.17%, F1 52.80%, 붕괴 없음 |
| 원본 전처리(0713_v2) | 43 | (미측정) | Acc 39.13%, F1 28.30%, **Prodromal 붕괴 재발** |

**결론: 완화되지만 완전한 해결은 아님.** 테스트한 4개 조합 중 3개는 붕괴가
사라졌지만, 1개(원본 전처리 + seed 43)는 보정을 적용해도 여전히 붕괴가
재현됨. 근본 원인(거대한 단일 Linear classifier + 212개뿐인 학습 샘플)은
논문이 지정한 구조라 구조 자체를 바꾸지 않는 한 완전히 없앨 수 없음 — 이번
조치는 "빈도를 낮추는 완화책"으로 이해해야 함.

## 5. 왜 정확도·F1이 함께 올라갔는가 (개선의 메커니즘)

붕괴가 없어졌다는 것과 숫자가 좋아졌다는 것은 별개 얘기라, 왜 이 둘이
같이 움직이는지 짚어둠.

1. **Macro 지표의 산술적 효과**: Precision/Recall/F1을 macro(클래스별 동일
   가중치) 평균으로 계산하기 때문에, 클래스 3개 중 1개가 0.00/0.00/0.00이면
   그 하나만으로 macro F1이 최대 1/3까지 깎임. 붕괴가 사라지는 것만으로도
   macro F1은 다른 두 클래스 성능이 그대로여도 **기계적으로 상승**함 —
   실제로 새 전처리 seed42 사례에서 F1이 46.76% → 61.15%로 뛴 것의 상당
   부분은 이 효과.
2. **학습 궤적 자체의 질적 변화**: 초기 로짓 분산을 정상 범위로 되돌리면
   학습 초반 손실이 극단적으로 크지 않은 상태에서 출발함. 손실이 처음부터
   폭주하면 Adam이 "한 클래스로 몰아 찍어 평균 손실을 빠르게 낮추는" 얕은
   지름길로 급격히 끌려가기 쉬운데, 완만한 손실 지형에서 시작하면 세
   클래스를 고르게 구분하는 방향으로 학습이 진행될 여지가 커짐 — 그 결과
   붕괴 안 한 클래스들의 recall/precision 자체도 대체로 더 균형 잡힘(예:
   새 전처리 seed42에서 Control 0.82, Prodromal 0.88 recall).
3. **Accuracy 상승도 같은 맥락**: 보정 전엔 무너진 클래스의 샘플이 전부
   오답 처리됐던 반면(예: Control 17명 전부 오답), 보정 후엔 그 클래스도
   부분적으로 맞히기 시작하면서 전체 정답 수가 늘어남.

**과대 해석 주의**: 이 개선은 "논문 목표(84.96%)에 근접했다"는 뜻이 아님.
여전히 붕괴가 재현된 조합(원본 전처리 seed43)이 있고, 근본 원인(작은
데이터셋 + 거대 classifier)은 그대로라 남은 격차는 이번 조치로 못 좁히는
구조적 한계로 봐야 함.

## 6. 시드를 여러 개로 늘려서 비교할 때 참고할 점

**장점**
- 시드 1~2개로 본 숫자는 "운"인지 "패턴"인지 구분이 안 됨. 여러 시드의 평균·
  표준편차를 내면 "새 전처리가 실제로 더 나은지"에 통계적 근거가 생김.
- 클래스 붕괴가 "가끔"인지 "거의 항상"인지 빈도로 파악 가능(현재는 4번 중
  1번 — 표본이 너무 적어 정확한 확률은 알 수 없음).

**주의할 점**
- `--seed`는 모델 초기화뿐 아니라 **train/val/test 분할 자체도 같이 바꿈**
  (`dataset.py`의 `get_holdout_split`이 동일 seed 사용). 즉 시드를 바꾸면
  "학습 불안정성"과 "어떤 피험자가 test set에 걸렸는가"라는 **두 가지 다른
  변동 요인이 함께 섞여서** 움직임 — 순수하게 학습 안정성만 보는 실험은 아님.
- Test set이 46명뿐이라 샘플 1명당 정확도가 약 2.17%p씩 흔들림. 시드를
  늘려도 이 노이즈 자체가 없어지지는 않고, 노이즈의 "크기"를 잴 수 있게
  될 뿐임.
